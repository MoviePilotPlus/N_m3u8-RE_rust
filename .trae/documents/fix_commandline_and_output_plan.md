# 修复命令行参数解析错误并复刻原项目输出格式计划

## 问题分析

### 1. 命令行参数解析错误
- Rust 项目在解析命令行参数时缺少 `skip-subtitle-decrypt` 等参数的定义
- 错误信息：`arg `skip-subtitle-decrypt`'s `ArgAction` should be one of `SetTrue`, `SetFalse` which should provide a default`

### 2. 输出格式不一致
- 当前 Rust 项目的输出格式与原项目不一致
- 需要实现 1:1 复刻原项目的输出格式

## 实施计划

### 第一部分：修复命令行参数解析

#### 步骤 1：添加缺失的命令行参数定义
- 在 `src/commandline/parser.rs` 文件中添加以下参数的定义：
  - `skip-subtitle-decrypt`：跳过字幕解密
  - `skip-audio-decrypt`：跳声音频解密
  - `force-mux-dolby`：强制混流杜比音效
  - `use-shaka-packager`：使用 Shaka Packager（已过时，建议使用 decryption-engine）
  - 确保所有参数的默认值与原项目一致

#### 步骤 2：修正参数处理逻辑
- 确保所有参数的处理逻辑与原项目一致
- 特别注意布尔类型参数的处理
- 确保 `-sv`、`-sa`、`-ss` 等缩写参数的处理正确

### 第二部分：分析并复刻原项目输出格式

#### 步骤 1：分析原项目输出格式
- 查看原项目的日志输出逻辑
- 分析输出信息的格式、颜色、结构等
- 特别注意不同阶段的输出信息

#### 步骤 2：实现输出格式复刻
- 修改 Rust 项目的日志输出逻辑
- 确保输出格式与原项目完全一致
- 包括启动信息、解析信息、下载进度、完成信息等

### 第三部分：测试验证

#### 步骤 1：编译测试
- 修复命令行参数解析错误后编译项目
- 确保编译成功

#### 步骤 2：功能测试
- 使用与原项目相同的命令行参数运行 Rust 版本
- 验证输出格式是否与原项目一致
- 验证功能是否正常工作

## 技术要点

### 命令行参数处理
- 使用 clap 库正确定义所有参数
- 确保参数的默认值与原项目一致
- 正确处理布尔类型参数和带值参数

### 输出格式复刻
- 分析原项目的日志输出模式
- 实现相同的输出结构和格式
- 确保颜色和样式与原项目一致

### 兼容性保证
- 确保所有命令行参数与原项目兼容
- 确保输出格式与原项目完全一致
- 确保功能行为与原项目一致

## 风险评估

### 潜在风险
- 命令行参数解析逻辑复杂，可能存在遗漏
- 输出格式复刻需要仔细分析原项目的输出模式
- 可能存在与原项目不同的行为差异

### 风险缓解措施
- 仔细对比原项目的命令行参数定义
- 详细分析原项目的输出格式
- 进行充分的测试验证

3. 输出格式与原项目完全一致
4. 所有功能正常工作
